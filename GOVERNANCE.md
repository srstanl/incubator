# Incubator Governance

## Entry Criteria

A project should move into `incubator` when most of the following are true:
- it has moved beyond casual experimentation
- it has a clear reason to exist
- it is worth structured investment
- it needs architecture, runtime, or platform decisions
- it has a realistic chance of graduating into a stronger asset
- it would benefit from explicit backlog tracking

## Expectations Inside Incubator

Projects here should develop:
- an intentional structure
- documented stack choices
- clear runtime boundaries
- CI expectations
- deployment assumptions
- explicit ownership of key decisions

At minimum, each incubated project should define:
- purpose
- current state
- target state
- key open questions
- graduation criteria

## Graduation Criteria

A project is ready to move to `portfolio` when it demonstrates most of the following:
- architecture is intentional rather than accidental
- stack choices are justified
- CI/CD expectations are in place or clearly defined
- documentation is coherent and current
- the project strengthens the portfolio narrative
- the project can serve as a reference or reusable asset

## Exit Criteria Without Graduation

A project may leave `incubator` without graduating if:
- the opportunity cost is too high
- the learning value has already been captured
- the project duplicates another stronger direction
- the project is useful, but not portfolio-relevant

In that case, it should either:
- return to `playground`
- remain a standalone repo with independent goals
- be archived with a short summary of lessons learned

## Board Guidance

Incubator projects should be tracked with a small execution board or equivalent backlog.

Recommended epic types:
- architecture and boundary cleanup
- runtime and packaging
- CI onboarding
- CD onboarding
- documentation and operating model

## Decision Rule

If a project is still asking “is this worth anything?” it belongs in `playground`.

If a project is asking “how do we shape this into something credible?” it belongs in `incubator`.

If a project is asking “how do we present and sustain this as a proven asset?” it belongs in `portfolio`.
